# gRPC 개념

- gRPC는 HTTP/2와 직렬화 기술을 사용하여 다음과 같은 특성을 가지고 REST API 대비 통신 성능 및 전송 데이터의 기밀성을 높인 RPC 방식이다.
- 전송되는 메세지의 양이 많아 질수록 REST API 대비 효율이 증가하므로 주로 MSA 구조에서 서비스간 통신에 사용된다.
  - 끊기지 않는 TCP 연결 (HTTP/2)
  - 양방향 통신 및 멀티 플렉싱 (HTTP/2)
  - 직렬화 데이터 구조 (protobuf)

## gRPC 연결 구조

- 클라이언트에서 channel(끊기지 않는 TCP 연결)을 생성하여 서버와 연결하고, channel을 기반으로 Stub 인스턴스를 생성한다.
- Stub은 grpc 서버의 Service에 대응되는 가벼운 객체로 하나의 client에서 여러 개 생성하여 사용할 수 있다.
- Stub은 Service에 정의된 rpc를 호출할 수 있는 객체이다.
- grpc 클라이언트 종료시 반드시 channel을 종료해 주어야 한다.(leak 방지)

### 요청 호출 흐름

> Client → Stub (요청 직렬화) → Channel (네트워크 전송) → Server (서비스 로직 처리)

- Service (서비스):
  - 서버가 제공할 원격 호출 함수들의 집합입니다.
  - Protocol Buffers(.proto) 파일에 정의되며, 클라이언트가 호출할 수 있는 메서드와 매개변수 구조를 명시합니다.

- Stub (스텁):
  - 클라이언트 측에서 원격 서버의 `서비스`를 로컬 함수처럼 쉽게 호출할 수 있게 해주는 가짜 객체(프록시)입니다.
  - 클라이언트의 요청을 네트워크로 전송 가능한 형태로 변환하고 서버의 응답을 다시 클라이언트 객체로 역직렬화하는 역할을 담당합니다.

- Channel (채널):
  - 클라이언트와 서버 사이의 실제 네트워크 연결(Connection) 상태를 관리하는 객체입니다.
  - HTTP/2 TCP 커넥션을 유지(백그라운드에서 살아있도록 계속 관리(Keep-Alive 핑을 주고받음)하는 역할)하고 로드 밸런싱을 처리하여, 스텁이 서버와 통신할 수 있는 파이프라인 역할을 합니다.
  - 클라이언트 종료시 반드시 channel 종료 필요

## gRPC 필수 개념

gRPC의 핵심 메커니즘을 완전히 이해하기 위해 반드시 알아야 할 5가지 필수 개념입니다.

### Protocol Buffers (Protobuf)

- 정의: 구글이 개발한 오픈소스 직렬화 데이터 구조입니다.
- 역할: gRPC의 IDL(인터페이스 정의 언어)이자 메시지 교환 포맷입니다.
- 특징: XML이나 JSON보다 크기가 작고 속도가 압도적으로 빠릅니다.

### 2. HTTP/2

- 정의: gRPC가 통신 기반으로 사용하는 네트워크 프로토콜입니다.
- 역할: 기존 HTTP/1.1의 성능 한계를 극복하는 하부 통신 레이어입니다.
- 특징: 한 커넥션에서 여러 요청을 동시에 처리하는 멀티플렉싱을 지원합니다.

### 4가지 통신 스트리밍 모델 (Streaming)

gRPC는 단순 요청-응답 외에 실시간 데이터 전송을 위한 스트리밍을 지원합니다.

- Unary: 단일 요청 -> 단일 응답 (기존 REST API 방식)
- Server Streaming: 단일 요청 -> 연속된 응답 스트림 (예: 실시간 뉴스 피드)
- Client Streaming: 연속된 요청 스트림 -> 단일 응답 (예: 파일 업로드)
- Bi-directional Streaming: 연속된 요청 -> 연속된 응답 (예: 실시간 채팅)

### Interceptor (인터셉터)

- 정의: 미들웨어와 유사한 개념으로, 요청과 응답을 중간에서 가로채는 기능입니다.
- 역할: 클라이언트와 서버 양측에서 공통 로직을 처리합니다.
- 용도: 로깅, 인증/인가, 모니터링, 메트릭 수집 등에 필수적입니다.

### Metadata (메타데이터)

- 정의: 일반적인 비즈니스 데이터가 아닌, 통신 자체에 대한 부가 정보입니다.
- 역할: HTTP 헤더와 유사하게 키-값(Key-Value) 형태로 전달됩니다.
- 용도: 인증 토큰(JWT), 요청 ID(추적용), 데드라인 등을 전송할 때 사용합니다.

gRPC 아키텍처 설계를 더 깊게 다루고 싶으시다면 다음을 고려해야한다.

- 마이크로서비스 간의 인증 및 보안(TLS/Token) 설정 방법
- 시스템 안정성을 위한 데드라인(Deadline) 및 타임아웃 처리
- 서비스 확장성을 위한 gRPC 로드 밸런싱(Load Balancing) 전략
