# Regex: Start/End of match (Match Limiters)

## `[[:alpha:]]-[[:alpha:]]` 패턴을 찾아서 모든 `-`를 `+`로 바꾸는 치환 명령

### Start/End of match 사용 (추천)

```text
:%s/[[:alpha:]]\zs-\ze[[:alpha:]]/+/g
```

- `\zs` (Start of match): 이 기호 이후부터 검색 결과로 간주 (후방 탐색 대용)
- 예: `ID:\zs\d\+` (ID: 뒤의 숫자만 선택)
- `\ze` (End of match): 이 기호 이전까지만 검색 결과로 간주 (전방 탐색 대용)
- 예: `\d\+\zepx` (px 앞의 숫자만 선택)

### Look-around 문법 사용

```text
:%s/\([[:alpha:]]\)\@<=- \@=[[:alpha:]]/+/g
```

Vim(vi)에서 전방 탐색(Look-ahead)과 후방 탐색(Look-behind)은 특정 문자를 찾을 때 '앞뒤 문맥'은 확인하되, 그 문맥 자체는 검색 결과(치환 대상)에 포함하지 않으려고 할 때 사용합니다.

Vim의 문법은 일반적인 Perl/Python 정규표현식(`(?=...)`, `(?<=...)`)과는 다릅니다.

#### 1. 전방 탐색 (Look-ahead): `\@=`

"다음에 특정 패턴이 오는 경우"를 검색합니다.

- 긍정 전방 탐색 (`\@=`): A 뒤에 B가 오는 경우의 A만 찾기
  - 예: `apple\@=pie` → applepie에서 apple만 선택
- 부정 전방 탐색 (`\@!`): A 뒤에 B가 오지 않는 경우의 A만 찾기
  - 예: `apple\@!pie` → applejuice의 apple은 찾지만, applepie는 제외

#### 2. 후방 탐색 (Look-behind): `\@<=`

"앞에 특정 패턴이 있는 경우"를 검색합니다.

- 긍정 후방 탐색 (`\@<=`): A 뒤에 오는 B 중 B만 찾기
  - 예: `\(apple\)\@<=pie` → applepie에서 pie만 선택
- 부정 후방 탐색 (`\@<!`): 앞에 A가 없는 경우의 B만 찾기
  - 예: `\(apple\)\@<!pie` → cherrypie의 pie는 찾지만, applepie는 제외

#### 3. 실전 예시 (치환 작업 시)

   1. 특정 단어 뒤의 숫자만 바꾸고 싶을 때
      - ID:123에서 ID:는 두고 123만 999로 치환
         - `:%s/\(ID:\)\@<=\d\+/999/g`
   2. 단위 기호 앞의 숫자만 선택하고 싶을 때
      - 100px, 200px에서 px 앞의 숫자만 찾기
         - `/\d\+\(px\)\@=`

#### perl/python Look-arounc(전후방탐색)

- 긍정형 후방 탐색: `(?<=pattern)` 후방에 pattern이 오는 경우에만 일치
- 부정형 후방 탐색: `(?<!pattern)` 후방에 pattern이 오지 않는 경우에만 일치
- 긍정형 전방 탐색: `(?=pattern)` 전방에 pattern이 오는 경우에만 일치
- 부정형 전방 탐색: `(?!pattern)` 전방에 pattern이 오지 않는 경우에만 일치

## `Start/End of match`와 `Look-around` 비교

Vim에서 `start/end of match (\zs, \ze)`와 `Look-around(@<=, @=)`는 모두 "주변 문맥은 보되, 매칭 결과(치환 대상)에서는 제외한다"는 목적은 같지만, 작동 방식과 성능에서 차이가 납니다.
쉽게 비유하자면, 경계 지정자는 '가위질할 범위를 지정'하는 것이고, Look-around는 '조건을 검사'하는 것입니다.

### 1. 작동 방식의 차이

| 구분 | Start/End of Match (`\zs`, `\ze`) | Look-around (`\@<=`, `\@=`) |
|---|---|---|
| 핵심 개념 | 검색 엔진의 포인터 이동 | 조건부 검사 (Assert) |
| 처리 방식 | 패턴 전체를 순서대로 훑다가 `\zs`를 만나면 그때부터 '진짜 매칭'으로 간주함. | 해당 위치에서 특정 패턴이 있는지 '미리 보거나' '뒤돌아본' 후, 다시 원래 자리로 돌아와 검색함. |
| 가독성 | 매우 직관적이고 단순함. | 괄호와 기호가 섞여 복잡함. |
| 성능 | 빠름. Vim에 최적화된 방식. | 상대적으로 느림. (특히 후방 탐색 `\@<=`) |

### 2. 주요 차이점 상세

- 후방 탐색의 한계 (Look-behind)
  - Look-around (`\@<=`): 이론적으로 앞에 오는 패턴의 길이에 제한이 없습니다. 하지만 패턴이 복잡해질수록 검색 속도가 급격히 느려집니다.
  - 경계 지정자 (`\zs`): 단순히 "이 지점부터"라고 선언하는 것이기 때문에 복잡한 패턴 뒤에서도 매우 빠르고 안정적으로 동작합니다.

- 중첩과 복합 조건
  `- Look-around: "앞에는 A가 있고, 동시에 뒤에는 B가 없어야 하며, 그 뒤에는 C가 있어야 한다"는 식의 다중 논리 조건을 걸기에 유리합니다.
  - 예: `\(A\)\@<=\d\+\(B\)\@!` (A 뒤에 오는 숫자이면서 동시에 뒤에 B는 없어야 함)
  - 경계 지정자: 한 매칭 안에 `\zs`와 `\ze`를 각각 한 번씩만 쓸 수 있어 복잡한 논리 구조를 만들기에는 한계가 있습니다.`

### 3. 실무적인 선택 기준

- 치환(`:s`) 작업 시: 무조건 경계 지정자(\zs, \ze)를 먼저 고려하세요. 타이핑이 훨씬 적고 오타 확률이 낮습니다.
- `:%s/Log: \zs\w\+/Success/g` (Log: 뒤의 단어만 교체)
- 복잡한 프로그래밍 패턴 검색 시: 검색 결과에는 포함되지 않아야 하지만, 동시에 여러 개의 부정 조건(Negative Look-ahead 등)을 따져야 할 때는 Look-around를 사용합니다.

결론적으로, Vim에서는 \zs와 \ze가 Vim만의 전매특허이자 강력한 무기이므로, 웬만한 상황에서는 이들을 사용하는 것이 정신 건강과 편집 속도에 이롭습니다.
이 두 방식을 사용하면서 매칭이 의도치 않게 길게 잡히거나 실패한 경우가 있었나요? 구체적인 사례를 주시면 해결책을 찾아드릴게요.
