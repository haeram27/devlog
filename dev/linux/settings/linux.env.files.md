# linux environment files

세 파일은 환경 변수가 적용되는 범위(모든 사용자 vs 특정 사용자)와 로드되는 시점(로그인 시 1회 vs 쉘 실행 시마다)에서 명확한 차이가 있습니다.
리눅스의 환경 변수 로드 순서와 각 파일의 핵심 차이점은 다음과 같습니다.

## 1. 한눈에 보는 비교표

| 파일 경로 | 적용 범위 | 로드 시점 | 파일 형식 | 주요 용도 |
|---|---|---|---|---|
| /etc/environment | 시스템 전체 (모든 사용자) | 로그인 시 1회 (PAM 모듈이 로드) | 단순 KEY=VALUE (스크립트 X) | 시스템 기본 언어(LANG), 전역 프록시 |
| /etc/profile.d/xxx.sh | 시스템 전체 (모든 사용자) | 로그인 쉘 시작 시 1회 | 쉘 스크립트 (.sh) | 특정 패키지(Java, Go 등)의 전역 경로 설정 |
| ~/.bashrc | 현재 사용자 전용 | 비로그인 대화형 쉘 실행 시마다 | 쉘 스크립트 | 개인 프록시, Alias(단축어), 프롬프트 커스텀 |

## 2. 파일별 상세 특징

### 📂 /etc/environment

- 동작 방식: 리눅스 로그인 프로세스 초기(PAM-PAM_env 모듈)에 읽힙니다.
- 특징: 쉘 스크립트 파일이 아님(`source` 명령 사용 불가능). 이 설정 파일은 `key=value` 형식의 설정만 가능. 따라서 파일내에서 export 명령어나 if 조건문, $PATH 같은 변수 치환을 사용할 수 없습니다.
- 주의점: 파일을 수정해도 이미 열려 있는 터미널이나 새로 여는 하위 쉘에는 반영되지 않습니다. WSL을 재시작(wsl --shutdown)하거나 로그아웃해야 적용됩니다.

### 📂 /etc/profile.d/xxx.sh

- 동작 방식: 사용자가 로그인할 때 /etc/profile 파일이 실행되면서, 내부적으로 /etc/profile.d/ 디렉토리 안에 있는 모든 .sh 스크립트 파일을 자동으로 불러와 실행합니다.
- 특징: 쉘 스크립트 파일(`source` 명령 사용 가능)이므로 export, if문 등을 사용할 수 있습니다.
- 주의점: 로그인 쉘(Login Shell)에서만 최초 1회 실행됩니다. GUI 환경에서 터미널 창을 새로 열거나, WSL에서 단순히 새 탭을 열 때는(비로그인 쉘) 다시 실행되지 않습니다.

### 📂 ~/.bashrc

- 동작 방식: 사용자가 새 터미널 창을 열거나 대화형 쉘(Interactive Non-Login Shell)을 실행할 때마다 매번 실행됩니다.
- 특징: 쉘 스크립트 파일(`source` 명령 사용 가능). 오직 해당 파일을 소유한 사용자 계정에만 영향을 미칩니다.
- 주의점: source ~/.bashrc 명령어로 재부팅 없이 수정한 내용을 즉시 현재 터미널에 반영할 수 있어 가장 유연합니다.

## 💡 WSL proxy 설정을 위한 최적의 선택 가이드

- 회사/학교 프록시망에서 WSL 내 모든 프로그램(전역)에 고정으로 적용하고 싶을 때: /etc/environment
- 개발 도구(Java, Go 등)를 수동 설치하고 모든 사용자의 PATH를 잡아주고 싶을 때: /etc/profile.d/my_env.sh 파일 생성
- 집과 회사로 이동하며 프록시를 껐다 켰다 하거나, 나만 프록시를 쓰고 싶을 때: ~/.bashrc

## `/etc/environment` 내용 현재 터미널 세션에 즉시 적용하기

`/etc/environment` 파일은 OS 재부팅 없이 현재 터미널 세션에 새로 변경된 설정을 즉시 적용할 수 있는 방법이 있습니다.

앞서 /etc/environment는 스크립트 파일이 아니기 때문에 단독으로 source 명령을 쓰면 안 된다고 말씀드렸지만, Bash의 특정 옵션이나 명령어 조합을 활용하면 현재 세션에 강제로 환경 변수로 로드할 수 있습니다.

현재 터미널 창(세션)에 즉시 반영하는 3가지 방법을 소개합니다.

### 방법 1. set -a 옵션과 source 조합 (가장 깔끔한 방법)

Bash 셸의 set -a (allexport) 옵션을 켜면, 이후 source되는 파일 안의 모든 Key=Value 형태의 일반 변수들이 자동으로 export 처리(환경 변수화) 됩니다.
터미널에 아래 명령어를 그대로 입력하면 됩니다.

```bash
set -a
source /etc/environment
set +a
```

- 작동 원리: `set -a`로 자동 export 기능을 켜고, 파일 내용을 현재 셸에 읽어들인 뒤(source), set +a로 기능을 다시 끕니다. 스크립트 파일이 아님에도 완벽하게 환경 변수로 등록하는 가장 널리 쓰이는 편법입니다.

### 방법 2. 원라인(One-liner) 루프문 사용

파일을 한 줄씩 읽어서 자동으로 export 명령어를 붙여 실행해 주는 셸 코드를 터미널에 바로 실행하는 방법입니다.

```bash
for env in $(cat /etc/environment); do export $(echo $env | sed -e 's/"//g'); done
```

- 작동 원리: 파일 내부의 문자열을 한 줄씩 가져와 따옴표(")를 제거한 뒤, export KEY=VALUE 형태로 현재 셸에 강제 등록합니다.

### 방법 3. export 문을 임시로 붙여 실행 (sed + eval)

파일 내용 앞부분에 export 라는 글자를 임시로 붙여서 가상의 스크립트처럼 바꾼 뒤, 현재 셸에서 평가(eval)해 실행하는 방식입니다.

```bash
eval $(sed 's/^/export /' /etc/environment)
```

### ⚠️ 주의사항 및 한계점

- 현재 터미널 창에만 적용: 위 방법들은 명령어를 입력한 '현재 터미널 창(세션)'에만 임시로 설정을 적용하는 것입니다. 이미 열려 있는 다른 터미널 창이나 시스템 백그라운드 서비스에는 반영되지 않습니다. 
- 시스템 전체(GUI, 다른 사용자 등) 완벽 반영: OS 재부팅을 하지 않더라도, 최소한 로그아웃 후 다시 로그인을 해야 리눅스의 PAM(pam_env.so) 모듈이 /etc/environment를 새로 읽어와 전체 시스템 세션에 정상 반영합니다.
