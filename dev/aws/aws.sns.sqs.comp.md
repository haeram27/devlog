# AWS SNS와 SQS 비교

AWS의 SNS(Simple Notification Service)와 SQS(Simple Queue Service)는 분산 시스템의 구성 요소를 분리(Decoupling)하는 메시징 서비스입니다.

핵심 차이는 SNS는 Pub/Sub(발행/구독) 기반으로 여러 수신자에게 메시지를 즉시 전달(PUSH)하고 메세지를 보관하지 않으며, SQS는 메시지 큐 방식의 풀(Pull) 방식으로 큐에 메시지를 저장(최소 1분, 최대 14일, 기본 4일)했다가 단일 소비자가 처리한다는 점입니다.

## 1. AWS SNS (Simple Notification Service)

- 방식: Pub/Sub (발행/구독), Push 기반
- 특징: 하나의 메시지를 여러 구독자(Lambda, SQS, HTTP, Email 등)에게 동시에 전달하는 팬아웃(Fan-out) 패턴에 특화되어 있습니다. 메시지는 즉시 전달되며, 저장되지 않습니다.
- 주요 용도:
- 실시간 알림 (사용자에게 SMS, 이메일, 모바일 푸시 발송)
   - 이벤트 기반 아키텍처: 하나의 이벤트를 여러 마이크로서비스에 동시 전파

## 2. AWS SQS (Simple Queue Service)

- 방식: Message Queue (큐잉), Pull 기반
- 특징: 메시지를 큐에 임시로 저장하여 소비자가 원하는 시점에 가져갑니다. 데이터 처리 속도 차이가 나는 시스템 간의 완충 장치(Buffering) 역할을 합니다.
- 주요 용도:
- 비동기 작업 처리: 주문 처리, 이미지 처리 등 시간이 걸리는 작업 분리
   - 부하 분산: 갑작스러운 트래픽 급증 시 큐에 메시지를 쌓아두고 백엔드 서비스 보호

## 요약 및 차이점 비교

| 구분 | SNS (Simple Notification Service) | SQS (Simple Queue Service) |
|---|---|---|
| 모델 | Pub/Sub (발행/구독) | Queue (큐) |
| 방식 | Push (즉시 전달) | Pull (소비자가 가져감) |
| 메시지 전달 | 1:N (여러 명에게 전달) | 1:1 (하나의 큐를 하나의 컴포넌트가 처리) |
| 지속성 | 메시지 미저장 (즉시 소멸) | 큐에 메시지 저장 (최대 14일) |
| 주요 목적 | 실시간 알림, 팬아웃 | 분산 시스템 간 디커플링, 부하 조절 |

보통 시스템에서는 SNS로 이벤트를 받아 여러 SQS 큐로 팬아웃하는 방식을 많이 사용합니다.
