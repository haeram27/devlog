S3Presigner API 시나리오 가이드

S3Presigner를 이용한 URL 발급 중심 아키텍처에서, 메서드 호출 플로우와 요청/응답 데이터 구조를 시퀀스 다이어그램으로 정리했습니다.

시나리오 1: 단일 PUT/GET 시나리오 2: 멀티파트 PUT/GET 시나리오 3: 삭제 위임 시나리오 4: 객체 메타 확인 시나리오 5: 버킷 접근 확인 시나리오 6: 멀티파트 실패 복구 시나리오 7: Range 다운로드

1. 단일 Object PUT/GET

사용 메서드: presignPutObject, presignGetObject

1) 클라이언트가 업로드/다운로드 요청
2) 백엔드가 S3Presigner로 presigned URL 발급
3) 클라이언트가 URL로 S3 직접 호출(서버 우회 데이터 전송)
서버 부하 감소 짧은 TTL 권장 헤더 정합성 필수
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: POST /files/presign-put\n{bucket,key,contentType,size} A->>P: presignPutObject(PutObjectPresignRequest) P-->>A: PresignedPutObjectRequest\n{url,signedHeaders,expiration} A-->>U: 200 OK\n{uploadUrl,method:PUT,headers,expiresAt} U->>S: PUT uploadUrl\n+ signed headers + file bytes S-->>U: 200 OK\n{ETag,VersionId?} U->>A: GET /files/presign-get?bucket&key A->>P: presignGetObject(GetObjectPresignRequest) P-->>A: PresignedGetObjectRequest\n{url,signedHeaders,expiration} A-->>U: 200 OK\n{downloadUrl,method:GET,expiresAt} U->>S: GET downloadUrl S-->>U: 200 OK\n{file stream}

요청/응답 데이터 (핵심)

단계 요청 데이터 응답 데이터
PUT URL 발급 bucket, key, contentType, contentLength, expiresIn uploadUrl, httpMethod=PUT, signedHeaders, expiresAt
파일 업로드 HTTP PUT + 파일 바이트 + 서명된 헤더 HTTP 200/204, ETag, VersionId(옵션)
GET URL 발급 bucket, key, response-content-type(옵션), expiresIn downloadUrl, httpMethod=GET, expiresAt
파일 다운로드 HTTP GET (presigned URL) HTTP 200 + 파일 스트림 
PUT presign API request 예시
{
  "bucket": "my-bucket",
  "key": "images/2026/05/a.png",
  "contentType": "image/png",
  "contentLength": 1048576,
  "expiresInSeconds": 300
}

PUT presign API response 예시
{
  "uploadUrl": "https://...",
  "method": "PUT",
  "headers": {
    "Content-Type": "image/png"
  },
  "expiresAt": "2026-05-12T04:30:00Z"
}
주의: presign 시 포함된 헤더(예: Content-Type, x-amz-server-side-encryption)는 실제 업로드 요청과 동일해야 합니다.

2. Multipart Upload PUT/GET

사용 메서드: presignCreateMultipartUpload, presignUploadPart, presignCompleteMultipartUpload, presignAbortMultipartUpload, presignGetObject

1) Multipart 시작 URL 발급 후 uploadId 획득
2) 파트별 presigned URL 발급 및 병렬 업로드
3) 수집한 ETag 목록으로 완료 요청
4) 필요 시 abort 호출, 완료 후 get URL 발급으로 다운로드
대용량 파일 병렬 전송 ETag 정합성 실패 시 Abort
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: POST /multipart/start\n{bucket,key,contentType} A->>P: presignCreateMultipartUpload(...) P-->>A: PresignedCreateMultipartUploadRequest\n{url,headers,expiration} A-->>U: {startUrl,expiresAt} U->>S: POST startUrl S-->>U: 200 OK\n{UploadId} loop each partNumber U->>A: POST /multipart/part-url\n{bucket,key,uploadId,partNumber} A->>P: presignUploadPart(...) P-->>A: PresignedUploadPartRequest\n{url,headers,expiration} A-->>U: {uploadPartUrl,partNumber} U->>S: PUT uploadPartUrl\n+ part bytes S-->>U: 200 OK\n{ETag} end U->>A: POST /multipart/complete\n{bucket,key,uploadId,parts:[{partNumber,eTag}]} A->>P: presignCompleteMultipartUpload(...) P-->>A: PresignedCompleteMultipartUploadRequest\n{url,headers} A-->>U: {completeUrl} U->>S: POST completeUrl\n+ XML(parts) S-->>U: 200 OK\n{Location,Bucket,Key,ETag} U->>A: GET /files/presign-get?bucket&key A->>P: presignGetObject(...) P-->>A: PresignedGetObjectRequest A-->>U: {downloadUrl} U->>S: GET downloadUrl S-->>U: 200 OK\n{file stream}

요청/응답 데이터 (핵심)

단계 요청 데이터 응답 데이터
Multipart 시작 URL 발급 bucket, key, contentType, expiresIn startUrl, expiresAt
Multipart 시작 호출 HTTP POST (startUrl) UploadId
파트 URL 발급 bucket, key, uploadId, partNumber, expiresIn uploadPartUrl, partNumber
파트 업로드 HTTP PUT + part bytes ETag (각 파트별 저장 필요)
완료 URL 발급 bucket, key, uploadId, parts[] completeUrl
완료 호출 HTTP POST + 파트 목록(XML/구조화 데이터) 완성 객체 메타(Location, ETag)
실패 시 중단 URL 발급 bucket, key, uploadId abortUrl 
multipart complete 요청 데이터 예시
{
  "bucket": "my-bucket",
  "key": "videos/2026/demo.mp4",
  "uploadId": "VXBsb2FkIElE...",
  "parts": [
    { "partNumber": 1, "eTag": "\"etag-part-1\"" },
    { "partNumber": 2, "eTag": "\"etag-part-2\"" }
  ]
}
권장: 파트 업로드는 병렬 처리하고, partNumber와 ETag를 영속 저장한 뒤 완료 단계에서 순서대로 제출합니다.
주의: multipart 중단 누락 시 미완료 파트 스토리지 비용이 누적됩니다. 실패 워크플로우에 presignAbortMultipartUpload를 포함하세요.

3. 객체 삭제 위임 (Delete)

사용 메서드: presignDeleteObject

1) 클라이언트가 삭제 요청
2) 백엔드가 삭제용 presigned URL 발급
3) 클라이언트가 URL로 S3에 직접 DELETE 요청
짧은 TTL 최소 권한 위임
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: POST /files/presign-delete\n{bucket,key} A->>P: presignDeleteObject(DeleteObjectPresignRequest) P-->>A: PresignedDeleteObjectRequest\n{url,expiration} A-->>U: 200 OK\n{deleteUrl,method:DELETE,expiresAt} U->>S: DELETE deleteUrl S-->>U: 204 No Content
요청 데이터 응답 데이터
bucket, key, expiresIn deleteUrl, httpMethod=DELETE, expiresAt

4. 객체 존재/메타 확인 (HEAD Object)

사용 메서드: presignHeadObject

1) 다운로드 전에 객체 존재/크기/ETag 확인 URL 발급
2) 클라이언트가 HEAD 호출로 메타만 수신
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: GET /files/presign-head-object?bucket&key A->>P: presignHeadObject(HeadObjectPresignRequest) P-->>A: PresignedHeadObjectRequest\n{url,expiration} A-->>U: {headUrl,method:HEAD,expiresAt} U->>S: HEAD headUrl S-->>U: 200 OK\n{Content-Length,ETag,Last-Modified}
요청 데이터 응답 데이터
bucket, key, ifMatch(옵션), expiresIn headUrl, httpMethod=HEAD, 메타 헤더(ETag, Content-Length)

5. 버킷 접근 가능 여부 확인 (HEAD Bucket)

사용 메서드: presignHeadBucket

1) 업로드 시작 전 백엔드가 head bucket URL 발급
2) 클라이언트가 HEAD 호출로 버킷 접근 가능 여부 확인
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: GET /bucket/presign-head?bucket A->>P: presignHeadBucket(HeadBucketPresignRequest) P-->>A: PresignedHeadBucketRequest\n{url,expiration} A-->>U: {headBucketUrl,method:HEAD,expiresAt} U->>S: HEAD headBucketUrl S-->>U: 200 OK or 403/404
주의: 403/404는 정책/존재 여부를 함께 반영할 수 있으므로, 클라이언트 메시지는 보안 관점에서 일반화하는 것이 좋습니다.

6. 멀티파트 실패 복구 (Abort 중심)

사용 메서드: presignAbortMultipartUpload

1) 파트 업로드 중 일부 실패/타임아웃 감지
2) 백엔드가 abort용 presigned URL 발급
3) 클라이언트가 abort 호출로 업로드 세션 정리
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: POST /multipart/abort-url\n{bucket,key,uploadId} A->>P: presignAbortMultipartUpload(AbortMultipartUploadPresignRequest) P-->>A: PresignedAbortMultipartUploadRequest\n{url,expiration} A-->>U: {abortUrl,method:DELETE,expiresAt} U->>S: DELETE abortUrl S-->>U: 204 No Content
요청 데이터 응답 데이터
bucket, key, uploadId, expiresIn abortUrl, httpMethod=DELETE, expiresAt
권장: 실패 재시도 횟수 초과 시 자동 abort를 실행해 미완료 파트 비용 누수를 방지합니다.

7. 대용량 부분 다운로드 (Range GET)

사용 메서드: presignGetObject (Range 헤더 포함)

1) 백엔드가 range 조건으로 presigned GET URL 발급
2) 클라이언트가 부분 범위만 다운로드해 재생 탐색/이어받기 구현
sequenceDiagram autonumber participant U as User/Client participant A as App Server participant P as S3Presigner participant S as Amazon S3 U->>A: GET /files/presign-get-range?bucket&key&start&end A->>P: presignGetObject(GetObjectPresignRequest with Range) P-->>A: PresignedGetObjectRequest\n{url,signedHeaders,expiration} A-->>U: {downloadUrl,method:GET,headers:{Range},expiresAt} U->>S: GET downloadUrl + Range bytes=start-end S-->>U: 206 Partial Content\n{Content-Range,partial bytes}
요청 데이터 응답 데이터
bucket, key, rangeStart, rangeEnd, expiresIn downloadUrl, headers.Range, 206 Partial Content
주의: Range 헤더를 서명에 포함한 경우 실제 호출 헤더 값이 달라지면 서명 검증에 실패합니다.
