# pfSense를 사용한 IP Gateway 생성 설정

pfSense를 활용하여 공인 IP망의 호스트가 사설망(192.168.x.x) 주소로 직접 접속할 수 있고, 사설망에서는 외부로 먼저 나갈 수 없도록 제어하는 가이드입니다.

pfSense를 사용하지 않는다면 리눅스의 경우 `net.ipv4.ip_forward=1` 커널 변수 설정(nic간 패킷 전달 허용)과 iptables 설정(FW 설정하여 LAN/WAN 상호간 )을  

인터페이스 명칭 매핑은 다음과 같습니다:

- eth0 = LAN (사설 IP 대역, 192.168.x.x)
- eth1 = WAN (공인 IP 대역)

------------------------------
## 1. pfSense 초기 인터페이스 및 IP 설정
pfSense 웹 GUI(또는 콘솔 메뉴)에서 물리 포트와 IP를 매핑합니다.

   1. Interfaces > Assignments에서 eth1을 WAN으로, eth0을 LAN으로 지정합니다.
   2. Interfaces > WAN에서 공인 IP 주소(Static IPv4)와 게이트웨이를 입력합니다.
   3. Interfaces > LAN에서 사설 IP 주소(예: 192.168.1.1/24)를 입력합니다.

------------------------------
## 2. pfSense 방화벽 및 NAT 설정 (가장 중요)
"외부 호스트가 사설 IP로 직접 들어올 수 있고 응답을 받되, 사설망 내부에서는 외부로 먼저 나갈 수 없는 환경"을 만드는 핵심 설정입니다.

### ① 사설망(LAN)의 아웃바운드(외부 행) 전면 차단

   1. Firewall > Rules > LAN 탭으로 이동합니다.
   2. 기본적으로 켜져 있는 Default allow LAN to any rule (IPv4 및 IPv6) 규칙 왼쪽에 있는 토글을 눌러 Disable(비활성화) 하거나 Delete(삭제) 합니다.
   3. 상단의 Apply Changes를 누릅니다. 이제 사설망 내부 기기들은 외부 인터넷으로 먼저 연결을 시작할 수 없습니다.

### ② 아웃바운드 NAT 해제 (사설 IP 원본 유지, outbound=LAN->WAN)
공인 IP망의 호스트가 사설망 서버의 실제 IP(192.168.x.x)를 목적지로 패킷을 보낼 때, pfSense가 중간에서 주소를 변환(NAT)하지 않고 원본 그대로 통과시켜야 정상적인 양방향 라우팅이 통신됩니다.

   1. Firewall > NAT > Outbound 탭으로 이동합니다.
   2. Mode를 Disable Outbound NAT (또는 Manual Outbound NAT 선택 후 기존 생성된 매스커레이딩 규칙 모두 삭제)로 변경합니다.
   3. Save 후 Apply Changes를 누릅니다.

### ③ 외부망(WAN)에서 사설망으로 들어오는 트래픽 허용
pfSense WAN 인터페이스는 기본적으로 사설 IP 대역을 차단하고, 외부에서 먼저 들어오는 트래픽을 모두 막습니다. 이를 해제해야 합니다.

   1. Interfaces > WAN 메뉴 가장 아래쪽에서 Block private networks and loopback addresses 체크박스를 해제하고 저장합니다.
   2. Firewall > Rules > WAN 탭으로 이동하여 Add (위쪽 화살표)를 눌러 규칙을 추가합니다:
   - Action: Pass
      - Protocol: TCP
      - Source: Any (또는 특정 외부 호스트 공인 IP)
      - Destination: LAN net (또는 사설망 특정 서버 IP 대역)
      - Destination Port Range: 접속을 허용할 특정 포트 범위 (예: From 8000 To 9000)
   3. Save 후 Apply Changes를 누릅니다.

------------------------------
## 3. 공인 IP망 호스트(클라이언트)에 추가해야 할 설정
공인 IP망에 있는 일반 컴퓨터(호스트)들은 192.168.x.x 패킷을 어디로 보내야 할지 모릅니다. 따라서 외부 호스트 컴퓨터에 "사설망으로 가려면 pfSense 외부 IP(WAN)를 거쳐서 가라"는 이정표(정적 라우팅)를 반드시 추가해야 합니다.

외부 호스트의 OS에 따라 아래 명령어를 관리자 권한으로 실행합니다. (※ pfSense의 WAN IP가 211.x.x.50이고, 사설망 대역이 192.168.1.0/24라고 가정)

### ① Windows 호스트인 경우
CMD(명령 프롬프트)를 관리자 권한으로 실행 후 입력:

```bash
route -p add 192.168.1.0 mask 255.255.255.0 211.x.x.50
route show
```

추후 삭제시
```bash
route show
route delete 192.168.1.0
```

- -p 옵션은 컴퓨터를 재부팅해도 라우팅 경로가 영구히 유지되도록 합니다.

### ② Linux 호스트인 경우
터미널에서 입력:

```bash
sudo ip route add 192.168.1.0/24 via 211.x.x.50
ip route show
```

추후 삭제시
```bash
ip route show
sudo ip route del 192.168.1.0/24
sudo route del -net 192.168.1.0 netmask 255.255.255.0
```

## 4. 최종 통신 흐름 요약

   1. 외부 호스트 ➡️ 사설망: 외부 호스트가 192.168.1.10:8000으로 접속 시도 ➡️ 추가한 route add 설정 덕분에 패킷이 pfSense WAN(eth1)으로 전달됨 ➡️ pfSense WAN 방화벽 규칙이 통과시킴 ➡️ LAN(eth0)을 통해 사설망 서버 도착.
   2. 사설망 ➡️ 외부 호스트 (응답): 사설망 서버가 외부 호스트로 답장 전송 ➡️ pfSense LAN으로 전달됨 ➡️ pfSense 방화벽의 State 테이블이 "외부에서 먼저 정상적으로 들어왔던 요청에 대한 답장 패킷"임을 인지하여 자동 허용 ➡️ 외부 호스트에 정상 응답 완료.
   3. 사설망 ➡️ 외부 (최초 연결 시도): 사설망 서버가 외부 구글(8.8.8.8) 등으로 먼저 접속 시도 ➡️ pfSense LAN 방화벽 규칙(Default allow 삭제)에 의해 전면 차단(DROP)됨.

## 공인 IP망(WAN) 호스트(클라이언트)에 추가한 route 설정 삭제 방법

### 1. Windows 호스트에서 라우팅 삭제 방법
CMD(명령 프롬프트) 또는 PowerShell을 반드시 관리자 권한으로 실행한 후 아래 명령어를 입력합니다.

- 특정 대역 라우팅 삭제:

```bash
route delete 192.168.1.0
```

- 참고: 영구 옵션(-p)을 주어 생성한 라우팅 규칙이라도 route delete 명령 한 번으로 영구히 삭제됩니다.
- 제대로 삭제되었는지 확인 (라우팅 테이블 조회):

```bash
route print
```

명령어 결과 화면의 'IPv4 경로 테이블(IPv4 Route Table)' 목록에서 192.168.1.0 대역이 사라졌는지 확인하면 됩니다.


### 2. Linux 호스트에서 라우팅 삭제 방법
터미널을 열고 sudo 권한을 사용하여 삭제 명령어를 입력합니다.

- 특정 대역 라우팅 삭제(ip 명령어):

```bash
sudo ip route del 192.168.1.0/24
```

- 만약 예전 방식의 route 명령어로 추가했었다면 아래 명령어로도 삭제할 수 있습니다:
```bash
sudo route del -net 192.168.1.0 netmask 255.255.255.0
```
   
- 제대로 삭제되었는지 확인 (라우팅 테이블 조회):

```bash
ip route show
```

목록에서 192.168.1.0/24 via ... 행이 보이지 않는다면 성공적으로 삭제된 것입니다.



## 참고: NAT 게이트웨이와 IP 게이트웨이(순수 라우터)의 차이

NAT 게이트웨이

- 의미: 패킷이 관문을 통과할 때 사설 IP를 공인 IP로 변환(숨김)해 주는 장치입니다. 
- 사용처: 가정용 공유기, 사설망 내 서버들의 인터넷 업데이트, 외부 노출을 숨겨야 하는 보안 네트워크에 주로 사용됩니다.

IP 게이트웨이 (순수 라우터)

- 의미: 주소 변환 없이 원본 IP를 그대로 유지하며 서로 다른 대역 간의 길만 찾아주는 장치입니다.
- 사용처: 본사-지사 간 내부망 연결(VPN), 회사 내 부서별 대역 통신, 외부와 실제 사설 IP로 직접 통신해야 하는 사내 인프라 구축에 사용됩니다.


| 구분 요소를 포함한 비교 항목 | NAT 게이트웨이 (주소 변환 방식) | IP 게이트웨이 (순수 라우터 방식) |
|---|---|---|
| 핵심 개념 (동작 원리) | 패킷이 통과할 때 IP 주소를 변환함. | IP 주소를 변경하지 않고 경로만 안내함. |
| 사설 IP의 외부 노출 여부 | 외부에서는 내부 사설 IP를 전혀 모름. | 외부 호스트가 사설 IP를 그대로 파악함. |
| 외부 호스트 접속 방식 | 게이트웨이 공인 IP와 포트 번호로 접속. | 서버의 실제 사설 IP(192.168.x.x)로 직접 접속. |
| 외부 호스트 설정 요구사항 | 추가 설정 필요 없음. | route add 명령어로 라우팅 추가 필수. |
| 패킷 주소 변형 (LAN ➡️ WAN) | 출발지 사설 IP를 게이트웨이 공인 IP로 바꿈. | 원본 사설 IP 주소를 그대로 유지하여 내보냄. |
| 가장 흔한 사용 예시 | 가정용 공유기, AWS NAT Gateway. | 기업 내 본사-지사 연결, 사내 내부망 간 통신. |


## 참고: SNAT과 DNAT의 이해

- SNAT (Outbound NAT)
  - LAN > WAN 패킷 전송에서 NAT GW가 패킷의 출발지 IP와 포트를 자신의 WAN IP와 임의 포트로 변환, 예: LAN의 호스트가 외부 인터넷 망에 요청을 보내고 응답을 받을수 있도록 NAT GW가 릴레이함
- DNAT (Inbound NAT)
  - WAN > LAN 패킷 전송에서 WAN의 호스트가 NAT GW로 보낸 패킷의 목적지 IP와 포트를 지정된 미리 지정된 LAN 호스트의 IP와 Port로 변환하여 패킷을 릴레이 함, NAT GW에는 GW의 WAN IP:port와 LAN 호스트 ip:port 맵핑 정보가 미리 설정되어 있어야 함(이를 포트포워딩 설정이라고 함)


### 1. 일반적인 NAT 게이트웨이 = SNAT만 활성화

- 방향: 내부(사설망) ➡️ 외부(인터넷)
- 동작: 내부 기기가 외부로 나갈 때, 출발지 주소(Source IP)를 게이트웨이의 공인 IP로 변환합니다.
- 특징: 외부에서 내부 사설망으로 먼저 접속하는 것은 불가능합니다. 오직 안에서 밖으로 나가는 통신만 허용됩니다.

### 2. 홈 네트워크 공유기 = SNAT + DNAT(선택) 결합 형태
가정이나 사무실에서 쓰는 일반 공유기는 NAT 게이트웨이의 일종이지만, 두 가지가 섞여 있습니다.

- 기본 상태 (SNAT 활성화): 모든 내부 PC가 인터넷을 쓸 수 있도록 출발지 IP를 변환합니다.
- 추가 설정 (DNAT / 포트 포워딩 활성화): 외부에서 내부 서버로 접속할 수 있도록 목적지 주소(Destination IP)를 사설 IP로 변환해 주는 규칙을 사용자가 수동으로 추가합니다.
