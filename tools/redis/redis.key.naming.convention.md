# redis key naming convention

## 핵심 네이밍 규칙

- 계층 구조 (구분자 - 콜론 `:` 사용): 가장 널리 사용되는 관례로, 콜론을 사용하여 데이터의 계층을 구분합니다.
  - 형식: [서비스/도메인(네임스페이스)]:[데이터타입]:[고유ID]:[속성]
  - 예시: user:profile:1001, order:status:20240306
  - 중요: **구분자는 임의로 선택한 문자를 사용해도 redis 사용에 아무런 영향이 없음. 단, 데이터 전체에 거쳐서 일관된 구분자 사용을 하는 것이 강력 권장됨.** redis와 연동하는 어플리케이션이나 솔루션 마다 가독성을 위해 기본적으로 인식하는 구분자는 있을 수 있음. 예:
    - `Redis Insight`, `Another Redis Desktop Manager`: 이 GUI 도구들은 `:`(콜론) 기준으로 key의 계층을 구분하여 디렉토리 형태로 화면에 보여줌
    - `Spring Data Redis`: 이 라이브러리는 기본값으로 `::`(더블 콜론)을 사용. `@Cachable` 등의 어노테이션 사용시 입력된 key 토큰을 `::` 구분자로 연결하여 생성함 (별도로 구분자 설정 가능)
- 소문자 사용: 대소문자를 구분하지만, 오타 방지와 가독성을 위해 소문자 사용을 강력히 권장합니다.
- 간결함과 의미 전달: 키가 너무 길면 메모리를 낭비하므로, 의미를 알 수 있는 선에서 최대한 짧게 구성합니다.
  - 나쁜 예: user_registration_details_id_1001
  - 좋은 예: user:1001:details

## 구성 요소별 가이드

- 네임스페이스 (Namespace): 키의 가장 앞에 위치하며, 앱 이름이나 환경(dev/prod) 등을 넣어 충돌을 방지합니다.
- **구분자** 선택: 일반적으로 콜론(:)을 사용하며, 단어 사이의 공백이 필요한 경우 스네이크 케이스(_)나 점(.)을 활용하기도 합니다.
- 동적 ID 포함: 데이터베이스의 PK나 세션 ID 등 고유 식별자를 포함하여 키의 유일성을 보장합니다.

## 네이밍 컨벤션 준수 시 장점

- 가독성: 키 이름만 보고도 어떤 데이터인지 즉시 파악 가능합니다.
- 디버깅 및 관리: Redis Insight 등 관리 도구에서 폴더 형태로 계층 구조를 시각화해주어 관리가 편리합니다.
- 패턴 매칭: SCAN 명령어 등을 사용하여 특정 그룹의 데이터를 한꺼번에 조회하거나 삭제하기 용이합니다.

## 주의 사항

- 너무 긴 키 이름: 키 자체도 메모리를 차지하므로 수천 바이트에 달하는 긴 이름은 피해야 합니다.
- 공백 및 특수 문자: SET "my key" 처럼 공백을 포함할 수 있으나, 프로그래밍 언어와의 호환성을 위해 피하는 것이 좋습니다.

## `Spring Data Redis`에서 key 값의 구분자(delimeter) 관례

### 왜 기본값이 `::` 인가?

- 표준화: Spring Cache 추상화에서 CacheName(그룹)과 Key(식별자)를 확실히 분리하기 위해 선택한 기본 규약입니다.
- 충돌 방지: 일반적인 데이터에 잘 쓰이지 않는 문자열을 사용하여, 키 값 자체에 `:`이 포함되어도 그룹명과 혼동되지 않도록 설계되었습니다.

### 설정 시 주의사항

    일관성: 운영 중인 서비스에서 구분자를 변경하면 기존에 저장된 캐시(`::` 사용)를 찾지 못하게 되므로, 서비스 초기 단계에서 설정하거나 기존 캐시를 비운 후(Flush) 변경해야 합니다.
    GUI 도구 호환성: 많은 Redis 관리 도구(Redis Insight 등)는 :을 기준으로 폴더를 자동 생성하므로, 시각적 편의를 위해 ::를 :로 바꿔서 사용하는 경우가 많습니다.


1. Redis 자체의 관습 (Standard Convention)
Redis 커뮤니티와 공식 문서에서 권장하는 전통적인 구분자는 :(싱글 콜론)입니다.

    Redis를 처음 만든 개발자들과 일반적인 NoSQL 설계 환경에서는 도메인:ID (예: user:100) 형식을 표준처럼 사용해 왔습니다.
    그래서 일반적인 CLI 명령어 예시나 설계 가이드에서는 :를 주로 사용합니다.

2. Spring Data Redis의 관례 (Framework Default)
반면, 질문하신 Spring Data Redis 라이브러리는 기본값으로 ::(더블 콜론)을 선택했습니다.

    이유: Spring은 '캐시 이름(value)'과 '키 값(key)'을 더 명확하게 분리하고 싶어 했습니다. 예를 들어, 키 값 자체에 이미 :이 포함되어 있는 데이터가 들어올 경우, 그룹명과 실제 키를 혼동하지 않기 위해 더 강력한 구분자인 ::를 채택한 것입니다.

요약하자면

    user:100: Redis의 전통적인 네이밍 방식 (CLI에서 직접 넣거나 다른 언어에서 쓸 때 흔함)
    user::100: Spring Cache (@Cacheable)를 사용할 때 생성되는 자동 생성 방식

따라서 Spring 프로젝트 내에서 CLI로 수동 데이터를 넣어 테스트하시려면, Spring의 규칙에 맞춰 SET "user::100" "value"와 같이 ::를 사용해야 프로그램이 해당 캐시를 인식할 수 있습니다.
